# Cackling Jack-o'-Lantern

**Source:** `pumpkin.asm`

A jack-o'-lantern on a black field that sits with a candle flickering inside it, then throws its head back and lets out an evil laugh — on power-on, when you press **RESET**, and when you press the joystick **fire** button. Like the [Christmas tree](../xmas/README.md), the whole pumpkin is a **reflected playfield** with no sprites, but this one adds a per-frame audio engine and a cycle-counted mid-scanline color kernel.

## Build & run

```sh
cd pumpkin
make            # assembles pumpkin.asm -> pumpkin.bin (+ .sym, .lst)
make run        # launches Stella on pumpkin.bin (needs Stella installed)
```

The build is `dasm pumpkin.asm -I../include -f3 -v0 ...`; `-I../include` points DASM at the shared headers (`vcs.h`, `macro.h`). It assembles to a 2 KB cartridge and produces a clean 262-line NTSC frame.

## The shape: a face made of holes

`CTRLPF` bit 0 turns on playfield reflection, so the right half of the screen mirrors the left and the pumpkin is symmetric for free — you only describe the left 20 columns.

```asm
    lda #%00000001         ; CTRLPF D0 = reflect
    sta CTRLPF
```

The trick to the face: **orange is where the playfield is ON; the eyes, nose, and mouth are gaps where it is OFF and the background color shows through.** Carving a jack-o'-lantern is literally cutting holes in it, and that is exactly what a playfield-OFF bit does here. The picture is drawn as a stack of horizontal "bands," each setting `PF0`/`PF1`/`PF2` in HBLANK and holding them for N scanlines. Because the bit order across `PF0`/`PF1`/`PF2` is irregular (see the Caveats), the bands are **generated from an ASCII design** rather than written by hand — `scratchpad/gen_pumpkin.py` turns art like this into the register values and their line counts:

```
   #####     ##############     #####     eyes-top   -> PF0=$80 PF1=$F0 PF2=$FE
```

Edit the art, regenerate, paste the bands back in. The mouth's teeth come out of the reflection: the two top teeth are one tooth per half (mirrored to two), and the single bottom tooth sits dead-center on the seam (`PF2` bit 7, mirrored to a 4-pixel tooth at the centerline).

## The light inside: candle glow & strobe

The face cutouts show the background color (`COLUBK`), so lighting them means changing `COLUBK`. Two moods share one mechanism:

- **Idle** — the cutouts flicker through warm gold/orange shades like a candle. A new shade is chosen at random every 8 frames from an 8-bit **LFSR** (the same one that seeds the laugh's warble), out of a palette kept deliberately brighter than the body orange so the features never blend in.
- **Laughing** — the cutouts flash hard white (`$0E`) in sync with the cackle, like a strobe light inside the pumpkin.

The hard part is that **a scanline has only one background color**, yet we want three regions: black sky, orange body, and a lit face — three colors from two. The solution is to change `COLUBK` *mid-scanline*, timed to the beam. A cycle-counted per-scanline kernel keeps `COLUBK` black at the screen edges and flips it to the glow color only across the body's interior (columns ~4–33), so the light shows through the eyes/nose/mouth but the sky beside the pumpkin never lights up:

```asm
.faceLine:
    sta WSYNC              ; cyc 0
    lda FacePF0,x
    sta PF0               ; ... playfield for this scanline (from a table)
    lda StrobeColor
    nop
    sta COLUBK            ; ~column 4:  glow ON, hidden under the body edge
    REPEAT 15
        nop               ; coast across the lit interior
    REPEND
    lda #0
    sta COLUBK            ; ~column 33: glow OFF, before the right-side sky
    dex
    bpl .faceLine
```

Because the body's left/right edges are a constant column 3/36 across the whole face, one fixed set of timings works for every line. The per-scanline `PF0/1/2` values come from generated `FacePF0/1/2` tables (page-aligned so the indexed reads don't drift the timing).

## The evil laugh

TIA can't play back a PCM sample while also drawing video — that path is 100% CPU and renders no picture (see `../music/`). So the laugh is a **frame-driven envelope player**: once per frame during VBLANK it walks a small table and writes the audio registers, which leaves the whole scanline budget for the picture. The table (`LaughData`, generated by `scratchpad/gen_laugh.py`) is a list of `(voice-pitch, packed-volume, frames)` rows:

```asm
LaughData:
    .byte  7,$0A,2,   7,$88,1,   8,$D5,1   ; MWAAA (breathe in -> falling wail)
    ...
    .byte  $FF                             ; end
```

Getting it to sound like a laugh rather than a *song* or a *drum* took some research into how the TIA has been used for vocal effects (the recipe is documented in the `LaughData` comment block in `pumpkin.asm`). The essentials:

- **Voice = a rough poly-tone**, not a pure tone — `AUDC 6`, the same timbre as *Adventure*'s "you got eaten" death sound. Pure tones read as melody.
- **Each "ha" is a big downward pitch sweep** with jittered, off-scale start pitches. Menacing laughter has a *wide* pitch range; landing on scale notes is what makes it sound like a tune. (`AUDF` is a divider, so sweeping a note *down* means *incrementing* `AUDF`.)
- **Aspirated "h" onset.** The volume byte packs two 4-bit volumes — voice in the high nibble, breath-noise in the low nibble — so a syllable can lead with a breath-only frame before the voice swells in. Without it, the tone slams on and sounds like a plosive "B".
- **A per-frame ±1 `AUDF` warble** (from the LFSR) gives a gravelly growl instead of a clean tone.

The three triggers (power-on, `RESET` via `SWCHB` bit 0, fire via `INPT4` bit 7) are edge-detected so one press is one laugh.

## Tips & Caveats

- **A `WSYNC` write one cycle too late costs a whole scanline.** The face kernel is a loop, and an early version's `sta WSYNC` started at cycle 74; since `sta` writes on its third cycle, the write landed at cycle 76 — the *next* line — so `WSYNC` halted an extra line and **every face line became two scanlines.** The frame ballooned to ~349 lines, Stella auto-detected it as **PAL** (wrong colors) and the forced-NTSC view flickered. The fix was to shave two cycles from the loop tail (`dex`/`bpl` instead of `inx`/`cpx`/`bne`) so the write lands by cycle 74. Moral: when a kernel loop drives `WSYNC`, count the cycles from the top `WSYNC` to the *next* one, and keep the write inside the 76-cycle line.
- **Green pumpkin = PAL palette, not a color bug.** The color bytes (`$38` orange, `$D4` green stem) are NTSC values; the same bytes in the PAL palette come out green and blue. If Stella shows the wrong colors, it has auto-detected the wrong TV format — which, per the point above, usually means the frame isn't a clean 262 lines.
- **Every band's line count is part of the 192-line budget.** The visible region (top margin + stem + caps + the 86-line face kernel + tapers + bottom margin) must total 192, and VSYNC 3 + VBLANK 37 + visible 192 + overscan 30 = 262. See [The Frame Structure](https://cdeever.github.io/atari-vcs/docs/tia-racing-the-beam/frame-structure/) in the book.
- **The generators are the source of truth.** The band table, the per-scanline face tables, the candle palette, and the laugh table are all emitted by the small Python scripts in `scratchpad/`. Edit the design there and regenerate rather than hand-editing the hex.
